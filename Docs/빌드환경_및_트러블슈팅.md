## TL;DR — 빌드는 반드시 VS2022로

이 PC에는 Visual Studio가 **두 개** 깔려 있다.

| 설치 | 경로 | CUDA 12.9 통합 |
|---|---|---|
| **VS2022 (v170)** | `...\Microsoft Visual Studio\2022\Community` | ✅ 있음 |
| VS2026 (v180) | `...\Microsoft Visual Studio\18\Community` | ❌ 없음 |

**→ 이 프로젝트는 VS2022 솔루션으로 빌드해야 한다.** VS2026이나 커맨드라인 `nvcc`로 빌드하면 컴파일러가 죽는다(아래 ①). 헷갈리는 이유는 `vswhere -latest`가 VS2026을 먼저 반환하기 때문이다.

검증된 빌드:

```powershell
# VS2022 개발자 환경 로드 후 MSBuild
call "C:\Program Files\Microsoft Visual Studio\2022\Community\VC\Auxiliary\Build\vcvars64.bat"
msbuild "D:\Github\RayTracinginOneWeekendinCUDA\RayTracinginOneWeekendinCUDA.sln" /t:Build /p:Configuration=Debug /p:Platform=x64 /m
```

성공하면 `x64\Debug\RayTracinginOneWeekend.exe`가 생기고, 실행하면 `output.ppm`이 저장된다. (실제로 빌드·실행 확인: scene 0 렌더 GPU 약 1.6초, `output.ppm` 12MB.)

> 가장 편한 방법은 그냥 **VS2022에서 `.sln`을 열고 F7(빌드)/F5(실행)** 하는 것이다.

## ① `cudafe++` ACCESS_VIOLATION — VS2026/커맨드라인 nvcc

**증상**

```
nvcc error : 'cudafe++' died with status 0xC0000005 (ACCESS_VIOLATION)
```

또는 VS2026 MSBuild에서:

```
error MSB4019: 가져온 프로젝트 "...\VC\v180\BuildCustomizations\CUDA 12.9.props"을(를) 찾을 수 없습니다.
```

**원인**

- CUDA 12.9 build customization(`CUDA 12.9.props`/`.targets`)은 **VS2022의 `MSBuild\Microsoft\VC\v170\BuildCustomizations`에만** 설치돼 있다. VS2026(v180)에는 없다.
- 커맨드라인 `nvcc`가 VS2026의 `cl.exe`(새 MSVC 헤더)를 호스트 컴파일러로 잡으면, CUDA 12.9의 `cudafe++`(디바이스/호스트 분리 프런트엔드)가 그 헤더를 파싱하다가 죽는다.
- **중요**: 이 크래시는 **원본 코드(텍스처 적용 전)로도 동일하게 재현**된다. 즉 우리 코드의 문제가 아니라 **툴체인 조합(VS2026 + CUDA 12.9)** 의 비호환이다.

**해결**

- **VS2022로 빌드**한다(위 TL;DR). 커맨드라인이 꼭 필요하면 VS2022의 `vcvars64.bat`를 먼저 로드해 v170 `cl.exe`를 쓰게 한다.

> 💡 진단 팁: 같은 명령으로 **원본 코드도 크래시하는지** 먼저 확인하면, 코드 문제인지 환경 문제인지 빠르게 가른다. `git stash -u` → 원본 빌드 → `stash pop`.

## ② stb_image를 `.cu`에 넣으면 cudafe++가 죽는다 (이미지 텍스처)

**증상**: `RtwImage.h`에서 `#define STB_IMAGE_IMPLEMENTATION` 후 `stb_image.h`를 include하고 그 헤더를 `kernel.cu`(.cu)에서 쓰면, 위 ①과 같은 `cudafe++` ACCESS_VIOLATION이 난다. 심지어 **선언부만** include해도 같은 TU에서 불안정했다.

**원인**: stb_image.h의 구현부는 약 5000줄의 순수 호스트 C 코드다. `.cu` 번역 단위에 들어가면 `cudafe++`가 이 거대한 코드를 파싱하다 죽는다.

**해결 (3단)**

1. **구현부 분리** — `StbImageImpl.cpp`(호스트 전용 `.cpp`, **cl이 컴파일**)에서만 `STB_IMAGE_IMPLEMENTATION`을 정의해 빌드한다. nvcc 디바이스 경로를 타지 않는다.
2. **`.cu`에는 전방 선언만** — `RtwImage.h`는 `stb_image.h`를 아예 include하지 않고, 쓰는 두 함수만 `extern "C"`로 전방 선언한다. 정의는 링크 시 `StbImageImpl.cpp`와 연결.

   ```cpp
   extern "C" {
       float* stbi_loadf(char const* filename, int* x, int* y,
                         int* channels_in_file, int desired_channels);
       void   stbi_image_free(void* retval_from_stbi_load);
   }
   ```
3. **vcxproj 등록** — `<ClCompile Include="StbImageImpl.cpp" />` (cl이 빌드), 헤더는 `<ClInclude>`.

> 📌 역할 분담: `StbImageImpl.cpp`(stb 구현부, 호스트) ↔ `RtwImage.h`(전방 선언 + 디바이스 업로드 로직, `.cu`에서 include).

## ③ 한글(멀티바이트) 주석 + stb 헤더 = 파서 desync

**증상**: `RtwImage.h`에서 한글 주석이 있으면 EDG 프런트엔드의 렉서가 줄바꿈을 삼켜 구조가 깨지는 파스 에러가 났다(`expected a declaration`, `identifier ... is undefined`, nvcc가 소스 라인을 2줄 적게 셈).

**원인**: UTF-8(BOM 없음) 한글이 시스템 코드페이지(cp949)로 오인식될 때, stb 헤더가 같이 끼는 특정 조합에서 렉서가 어긋났다. (다른 헤더의 한글 주석은 멀쩡히 컴파일되므로 일반적인 문제는 아니다.)

**해결**

- **`RtwImage.h`의 주석만 ASCII(영어)** 로 유지한다(stb와 직접 엮이는 파일이라). 나머지 헤더(`Texture.h`, `Perlin.h` 등)는 한글 주석 그대로 둔다.
- BOM 추가는 **금물** — `.cu`가 include하는 헤더에 BOM이 있으면 `cudafe++`가 또 다른 ACCESS_VIOLATION으로 죽는다. (시도해봤고 더 나빠진다.)

> C4819 경고("현재 코드 페이지(949)에서 표시할 수 없는 문자")는 한글 주석 때문에 **대량으로 뜨지만 무해**하다. 빌드는 정상 통과한다. 굳이 없애려면 모든 소스를 UTF-8 BOM으로 저장하거나 `/utf-8`을 줘야 하는데, 위 ②③ 때문에 권장하지 않는다.

## 장면 전환 방법 (Texture / Perlin 결과 보기)

월드는 GPU 커널(`CreateWorld`)에서 만들어지고, `main()`의 `sceneId` 한 줄로 어떤 장면을 렌더할지 고른다.

```cpp
// kernel.cu, main()
int sceneId = 0;
//   0: bouncing_spheres  — 바닥이 체커 텍스처인 최종 랜덤 구 장면 (기본)
//   1: checkered_spheres — 체커 구 2개
//   2: earth             — 지구 이미지 텍스처 구 (earthmap.jpg 필요)
//   3: perlin_spheres    — 펄린 노이즈(대리석) 구 2개
```

- **scene 2(지구)** 는 실행 디렉터리에 `earthmap.jpg`가 있어야 한다. 없으면 큰 **청록색** 구로 표시된다(디버깅용 fallback). VS에서 실행 시 작업 디렉터리는 보통 `$(ProjectDir)`(= `RayTracinginOneWeekend\`)이므로 그 위치에 이미지를 둔다.
- scene 0/1/3은 절차적 텍스처라 외부 파일이 필요 없다.

## 기타 무해한 경고들

| 경고 | 의미 | 조치 |
|---|---|---|
| `warning C4819` (다수) | 한글 주석 코드페이지 | 무시 (위 ③) |
| `ptxas warning : Stack size ... cannot be statically determined` | 재귀/가상함수 때문에 정적 스택 추정 불가 | 무시 (런타임에 `cudaLimitStackSize`로 충분히 잡음) |
| `LNK4098: 'LIBCMT' ... 충돌` | stb의 기본 런타임 lib와 충돌 | 무시 (정상 링크됨) |
| `nvcc warning : Support for ... prior to sm_75 ... removed in future` | 구형 아키텍처 deprecation | 무시 (또는 타깃 아키텍처 상향) |
